## 3. 기술 리스크

| 리스크 | 가능성 | 영향 | 완화 방안 | Plan B |
|--------|--------|------|----------|--------|
| 지도 API 무료 호출 한도 초과 | low | medium | 개인 사용이므로 일일 호출량이 낮을 것이나, API 응답을 로컬에 10분간 캐싱하여 중복 호출을 줄인다. | Google Places API 등 대체 API로 전환하거나, 캐싱 기간을 늘린다. |
| 위치 정확도 부족으로 주변 식당을 제대로 탐색하지 못함 | medium | medium | GPS 위치와 네트워크 위치를 병합 사용하고, 위치 정확도가 낮을 때 사용자에게 알린다. | 수동 주소 입력 기능을 폴백으로 제공한다. |

## 4. 되돌리기 어려운 결정

| 결정 | 왜 되돌리기 어려운가 | 확신도 | 되돌림 비용 |
|------|-------------------|--------|-----------|
| Flutter로 크로스플랫폼 앱 개발 | Flutter 기반으로 UI와 비즈니스 로직을 모두 작성하면, 네이티브 Swift/Kotlin으로 전환 시 전체 재개발이 필요하다. | high | iOS/Android 각각 네이티브 앱을 새로 개발해야 하며, 수주~수개월의 재작업이 필요하다. |
| 외부 지도 API 의존 (자체 식당 DB 미구축) | 자체 DB 없이 외부 API에 의존하면, 해당 API 서비스가 종료되거나 정책이 변경될 때 대응이 어렵다. | medium | 자체 크롤링 또는 다른 API로 전환 시 데이터 수집 및 파이프라인 구축에 상당한 시간이 소요된다. |
