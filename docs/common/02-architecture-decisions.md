## 5. ADR 로그


### ADR-1: 아키텍처 패턴

- **결정**: monolith
- **맥락**: 1인 사용자를 위한 단일 모바일 앱으로, 별도 백엔드 서버가 필요 없으며 Flutter 단일 앱 내에서 모든 로직을 처리하는 것이 가장 단순하고 빠르다.
- **검토한 대안**:

| 대안 | 장점 | 단점 | 탈락 사유 |
|------|------|------|----------|
| microservices | 서비스별 독립 배포 가능 | 1인 개인 앱에 과도한 복잡성 | 사용자가 1명이고 배포 인프라 관리 부담이 불필요하다. |
| modular_monolith | 모듈 간 경계가 명확하여 유지보수 용이 | 단일 앱 치고는 구조가 복잡할 수 있음 | 초기 빠른 개발 목표에는 단순한 모놀리스가 더 적합하다. |

- **상태**: 확정

### ADR-4: 지도 API 서비스 선택

- **결정**: 카카오 로컬 API (Kakao Local API)
- **맥락**: 주변 식당 검색을 위한 지도 API가 필요하며, 한국 내 식당 데이터의 정확성과 커버리지가 핵심 기준이다. 카카오 로컬 API는 카카오맵 기반의 풍부한 국내 POI 데이터를 제공하고, 키워드/카테고리 기반 검색과 거리순 정렬을 기본 지원한다.
- **검토한 대안**:

| 대안 | 장점 | 단점 | 탈락 사유 |
|------|------|------|----------|
| Google Places API | 글로벌 커버리지, 풍부한 부가 정보 (리뷰, 사진) | 국내 식당 데이터가 카카오/네이버 대비 부족, 비용이 비쌈 | 한국 식당 검색 특화 앱에서 국내 데이터 품질이 낮고 비용 부담이 크다. |
| 네이버 지도 API | 국내 데이터 풍부, 네이버 플레이스 연동 | API 사용 제한이 엄격, 모바일 앱 연동 절차가 복잡 | 카카오 대비 API 접근성이 낮고, REST API 키워드 검색 기능이 제한적이다. |

- **상태**: 확정
